PRD: Proactive Heartbeats (Autonomous Agent Scheduling)

Generated by: Milo Date: 2026-04-11 Status: Draft Requested by: Z

1. Goal

Make each of the 7 AI team members proactive -- they check in on their domain every few hours, surface new information, flag risks, and bring status updates to the user without being asked. Like real employees who show up with work done, not just respond when poked.

2. Why

Right now, a nonprofit leader has to know what to ask. If they don't think to check on grants, grants get missed. If they forget a funder deadline, it slips. Proactive heartbeats flip the dynamic: the team works in the background and brings updates to the leader. This makes Edify OS feel like having actual staff, not a chatbot.

Who it's for: Every Edify OS user. This is especially valuable for small nonprofits where one person wears many hats and can't stay on top of everything.

Core insight: The difference between a tool and a teammate is that a teammate doesn't wait to be asked.

3. Success Criteria

  • [ ] Each of the 7 archetypes has a configurable heartbeat schedule (default: every 4 hours during business hours)
  • [ ] Each heartbeat scans the archetype's domain using org memory + connected integrations
  • [ ] Heartbeats produce concise status updates delivered to the user's inbox/dashboard
  • [ ] Users can configure per-archetype: frequency, active hours, what to scan for, notification preferences
  • [ ] Heartbeats respect quiet hours (no notifications outside configured window)
  • [ ] Each update includes: what was found, why it matters, suggested action
  • [ ] Users can respond to a heartbeat update to start a conversation with that team member
  • [ ] Heartbeat history is viewable per archetype
  • [ ] Cost per heartbeat cycle (all 7 archetypes): under $0.05 on Qwen
  • [ ] System handles orgs in different timezones correctly

4. Tech Stack

  • Scheduler: Cron-based task scheduler (Python, celery-beat or APScheduler)
  • Backend: Python (existing agent service)
  • LLM: Qwen-Plus default (platform key), BYOK optional
  • Database: Supabase/PostgreSQL (heartbeat config, heartbeat logs, notification queue)
  • Notifications: In-app inbox (existing), email digest (optional), push (future)
  • Frontend: Next.js (existing apps/web)

5. Scope -- What is IN

5.1 Per-Archetype Proactive Scans

Each archetype gets a "proactive scan" prompt that defines what it looks for autonomously:

Development Director: - Scan for new grant opportunities matching org mission (via grant DB integration or org memory) - Flag grants with upcoming deadlines (< 30 days) - Check donor engagement: any donors who haven't been contacted in 90+ days - Surface fundraising progress vs. annual target

Marketing Director: - Review recent campaign performance (if social integrations connected) - Suggest content calendar items for upcoming week - Flag brand voice inconsistencies in recent org communications (if uploaded) - Surface trending topics relevant to org mission

Executive Assistant: - Summarize unread/unprocessed emails (if Gmail connected) - Flag upcoming meetings that need prep (if Calendar connected) - Surface overdue action items from past conversations - Compile daily briefing

Programs Director: - Flag upcoming funder reporting deadlines - Check outcome data gaps (metrics that haven't been updated) - Surface program milestones approaching - Flag compliance items due within 14 days

Finance Director: - Calculate current cash runway (if financial data in memory) - Flag budget variances above threshold - Surface upcoming financial reporting deadlines - Alert on restricted fund spending approaching limits

HR & Volunteer Coordinator: - Flag volunteer roles that are understaffed - Surface upcoming policy review dates - Check for training/certification expirations - Summarize new volunteer applications (if tracked in memory)

Events Director: - Surface upcoming event milestones (countdown-based alerts) - Flag overdue event planning tasks - Check sponsorship pipeline status - Post-event follow-up reminders

5.2 Scheduling System

  • Per-archetype cron configuration stored in DB
  • Default schedule: every 4 hours, 8 AM - 8 PM in org's timezone
  • Configurable: frequency (1h, 2h, 4h, 8h, daily), active window, days of week
  • Org-level master toggle (pause all heartbeats)
  • Per-archetype toggle (pause individual team members)

5.3 Notification & Delivery

  • In-app inbox: Primary delivery channel. Updates appear in the existing dashboard inbox.
  • Email digest: Optional. Batches all heartbeat updates into a single daily email (morning or evening, user's choice).
  • Push notifications: Future -- not in this build.
  • Each notification: archetype name, summary title, body (2-3 sentences max), suggested action button, "Discuss" button to open conversation.

5.4 Configuration UI

  • Settings page: "Your Team's Schedule"
  • Per-archetype row: toggle on/off, frequency dropdown, active hours picker
  • Global: timezone selector, email digest toggle, quiet hours
  • Preview: "Your Development Director will check in at 8 AM, 12 PM, 4 PM, and 8 PM"

5.5 Heartbeat Execution Pipeline

  1. Scheduler triggers archetype heartbeat
  2. Load archetype's proactive scan prompt
  3. Retrieve domain-scoped org memory
  4. Check connected integrations for new data (email, calendar, social, grants)
  5. LLM generates update (max 500 tokens)
  6. If update contains actionable content: deliver to inbox + optional email
  7. If nothing noteworthy: skip (don't spam "nothing new")
  8. Log heartbeat execution (timestamp, archetype, delivered or skipped, token usage)

6. Scope -- What is OUT

  • Real-time streaming updates (heartbeats are periodic, not live)
  • Autonomous actions (heartbeats report and suggest, they don't execute)
  • Cross-archetype coordination (each heartbeat is independent)
  • SMS or phone notifications
  • Mobile push notifications
  • Custom scan prompts (user-defined) -- future
  • Webhook delivery to external systems

8. Key Decisions

Decision Choice Reason
Scheduler APScheduler (in-process) No extra infrastructure (no Redis/Celery), works for current scale
Default frequency Every 4 hours Useful without being spammy. 4x/day during business hours = 4 updates
Skip if nothing new Yes -- only deliver when there's something actionable Prevents notification fatigue, builds trust that updates matter
Notification channel In-app inbox primary, email digest optional Meets users where they already are
Autonomy level Report + suggest only, never execute Users must approve actions. Trust is earned.
Token budget 500 tokens per heartbeat scan Keeps cost at ~$0.002/scan on Qwen. Full cycle (7 archetypes) = ~$0.014
Timezone handling Org-level timezone stored in DB, scheduler converts Nonprofits operate in local time, not UTC

9. Cost Analysis

Per org, per day (default 4x/day schedule, all 7 archetypes): - 7 archetypes x 4 scans x ~2.5K tokens input + ~500 tokens output = ~80K tokens/day - On Qwen: ~$0.05/day per org = ~$1.50/month per org - On Claude: ~$0.50/day per org = ~$15/month per org

At 100 orgs on Qwen: ~$150/month total. Very manageable as platform cost.

  • DB additions: heartbeat_configs table (org_id, archetype, enabled, frequency, active_start, active_end, timezone), heartbeat_logs table (id, org_id, archetype, timestamp, delivered, skipped_reason, token_usage)

  • Migration: Default heartbeat config created for each org on first login (all 7 enabled, 4h frequency, 8AM-8PM)

11. Handoff Notes

  • The proactive scan prompts are the soul of this feature. They need to be specific enough to find real value but general enough to work across different nonprofits. Each prompt should start with "You are checking in on your domain. Scan org memory and connected integrations for anything the leader should know about."
  • The "skip if nothing new" logic is critical. A naive implementation that always delivers will kill trust fast. The LLM should be prompted to return a special token (e.g., "[NOTHING_NEW]") when there's genuinely nothing to report.
  • Email digest should group updates by archetype with clean formatting. Think of it as a morning briefing.
  • The scheduler must handle server restarts gracefully (persist next-run times in DB, catch up on missed beats).
  • Connected integrations (email, calendar, social) massively increase heartbeat value. Without them, heartbeats can only scan org memory. The onboarding flow should emphasize connecting accounts.
  • Rate limit consideration: 100 orgs x 7 archetypes x 4 scans = 2,800 LLM calls/day. On Qwen free tier (10 req/min), that's ~4.7 hours of sequential processing. Need batching or multiple API keys at scale.
