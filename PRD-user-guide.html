PRD: User Guide & Onboarding System

Generated by: Milo Date: 2026-04-10 Status: Draft Requested by: Z

1. Goal

Build a comprehensive user guide and in-app onboarding system that makes Edify OS immediately understandable to nonprofit leaders with zero technical background.

2. Why

Our users are nonprofit executive directors, program managers, and development officers -- not engineers. They've never used an AI agent platform. Without clear guidance, they'll either: - Underuse the tool (only use one archetype as a generic chatbot) - Get confused by the multi-agent concept and churn - Miss high-value features like Decision Lab and subagent delegation

The user guide needs to explain not just WHAT the tool does, but HOW to think about it. "You've just hired 7 AI specialists. Here's how to work with each one."

Who it's for: Every Edify OS user, especially first-time users during their first week.

3. Success Criteria

  • [ ] New user can understand the platform's value prop within 60 seconds of landing
  • [ ] In-app onboarding guides user through their first interaction with each archetype
  • [ ] Help center covers every feature with searchable, illustrated articles
  • [ ] Contextual tooltips appear on first encounter with any UI element
  • [ ] User guide explains Decision Lab with example scenarios and sample outputs
  • [ ] Guide uses nonprofit-native language (no jargon like "agent", "subagent", "LLM", "delegation")
  • [ ] Every article answers: What is this? Why would I use it? How do I use it? (3-part structure)
  • [ ] FAQ section addresses common concerns (privacy, accuracy, cost, AI limitations)
  • [ ] Onboarding completion rate trackable via analytics
  • [ ] Guide is maintainable -- content lives in markdown, rendered dynamically

4. Tech Stack

  • Content format: Markdown files (easy to update, version-controlled)
  • Rendering: Next.js pages with MDX support
  • Search: Client-side full-text search (Fuse.js or similar)
  • Onboarding: React state machine for guided flows
  • Tooltips: Existing tooltip library (Radix UI or similar)
  • Analytics: Track onboarding step completion, article views, search queries

5. Scope -- What is IN

5.1 In-App Onboarding Flow

  1. Welcome screen -- "You've just hired a team of 7 AI specialists" with visual team intro
  2. Pick your first hire -- user selects one archetype to start with (guided first conversation)
  3. Guided first interaction -- pre-filled prompt suggestions, explains what's happening as the agent responds
  4. Feature discovery -- progressive disclosure of Decision Lab, memory, and advanced features over first week
  5. Completion rewards -- checkmarks, progress bar ("You've worked with 3 of 7 team members")

5.2 Help Center / Knowledge Base

  1. Getting Started -- platform overview, key concepts, first steps
  2. Meet Your Team -- one article per archetype explaining personality, strengths, when to use them, example prompts
  3. Decision Lab Guide -- what it is, when to use it, walkthrough with screenshots, sample scenarios
  4. Working with Your Team -- how to give good instructions, how to review outputs, how to provide feedback
  5. Organization Setup -- connecting accounts (OAuth), uploading org info, customizing agent behavior
  6. FAQ -- privacy, security, accuracy, pricing, limitations
  7. Troubleshooting -- common issues and fixes

5.3 Live Chat Support

  1. In-app chat widget -- persistent chat button on every page, opens a support conversation
  2. Powered by Edify's own agents -- the Executive Assistant archetype handles first-line support using the org's help center content as context
  3. Escalation path -- if the agent can't resolve the issue, it flags for human support (email ticket)
  4. Chat history -- users can revisit past support conversations
  5. Proactive help -- agent detects when a user seems stuck (long idle on a page, repeated failed actions) and offers assistance

5.4 Contextual Help

  1. Tooltips -- first-time hover explanations on all UI elements
  2. Empty states -- helpful guidance when a section has no data yet
  3. Inline suggestions -- "Try asking your Development Director to..." style prompts
  4. Feature announcements -- in-app banner for new features

5.5 Language Guidelines

  • Never say: agent, subagent, LLM, model, token, API, delegation, orchestration
  • Instead say: team member, specialist, AI hire, assistant
  • Frame everything as hiring/managing a team, not using software
  • Use "your Development Director" not "the Development Director agent"
  • Keep reading level at grade 8 (Flesch-Kincaid)

6. Scope -- What is OUT

  • Video walkthroughs (requires human production -- future)
  • Community forum
  • Localization / multi-language support
  • Printed/PDF user manual
  • Developer documentation (API docs are separate)

8. Key Decisions

Decision Choice Reason
Content format Markdown with MDX Easy for non-engineers to update, version-controlled, renders beautifully
Search Client-side (Fuse.js) No server needed, fast, works offline
Onboarding style Progressive disclosure over 7 days Avoids overwhelming on day 1
Language level Grade 8 reading level Accessible to all nonprofit professionals
Metaphor "Hiring a team" not "using a tool" Makes AI agents intuitive -- everyone knows how to work with people
Tooltip trigger First visit only (dismissable) Helpful without being annoying
Article structure What / Why / How (3-part) Consistent, scannable, answers all questions
Analytics Track onboarding completion + article views Know where users drop off

9. Deployment Plan

  • Repo: whitmorelabs/edify-os
  • Deploy target: Same Vercel deployment
  • Content updates: Merge to main, auto-deploys
  • No new services or databases -- content is static markdown, onboarding state in localStorage

10. Handoff Notes

  • The onboarding flow should feel warm and personal -- not a corporate tutorial. Think "welcome to your new team" energy.
  • Each archetype's guide article should include 5-10 example prompts that users can copy-paste to get started
  • Decision Lab guide should include a full walkthrough with a realistic nonprofit scenario (e.g., "Should we expand our food bank to a second location?") showing all 7 archetype responses
  • The help center search should prioritize results by "most commonly viewed" as usage data builds up
  • Empty states are critical -- a user who opens their Finance Director for the first time should see helpful suggestions, not a blank chat
  • All content should be reviewed for nonprofit-specific language accuracy (we're not a startup accelerator)
